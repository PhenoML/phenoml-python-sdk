## 9.5.0 - 2026-04-02
* The SDK now retries failed requests on network-level errors (`ConnectError`, `RemoteProtocolError`) using exponential backoff in addition to retrying on HTTP 5xx/429/408/409 responses. A new `ParsingError` exception is raised when the SDK receives a response that cannot be deserialized, providing clearer diagnostics. The minimum supported Python version has been raised to **3.10**; users on Python 3.8 or 3.9 must upgrade their Python runtime to continue using this SDK.
* The SDK now raises a new `ParsingError` exception (available from `phenoml.core`) when a server response cannot be validated against the expected schema. This replaces previously unhandled Pydantic `ValidationError` exceptions and provides structured access to the HTTP status code, response headers, raw body, and the underlying cause. Consumers can catch `ParsingError` to handle unexpected response shapes gracefully.
* The SDK now raises a `ParsingError` (instead of an unhandled `pydantic.ValidationError`) when a response from the API cannot be deserialized into the expected model. This provides callers with structured error context — including the HTTP status code, response headers, raw body, and the underlying cause — consistent with other SDK error types.
* The SDK now raises a `ParsingError` instead of an unhandled Pydantic `ValidationError` when a server response cannot be deserialized in the workflows client. This provides a consistent error type with full response context (status code, headers, and body) for all workflow API calls.
